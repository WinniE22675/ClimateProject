# สถาปัตยกรรมระบบหลังบ้าน (Backend Architecture)

ระบบหลังบ้านของโปรเจค **Climate Data Services** พัฒนาขึ้นด้วย **FastAPI** ทำหน้าที่เป็นศูนย์กลางประมวลผลประสิทธิภาพสูงแบบอะซิงโครนัส (Asynchronous) สำหรับจัดการชุดข้อมูลสภาพภูมิอากาศแบบหลายมิติ (NetCDF) สถาปัตยกรรมถูกออกแบบโดยเน้นการแยกสัดส่วนหน้าที่อย่างชัดเจน เพื่อการจัดการหน่วยความจำที่มีประสิทธิภาพเมื่อต้องรับมือกับข้อมูลขนาดใหญ่ และการคำนวณทางภูมิสารสนเทศที่แม่นยำ

### เทคโนโลยีและไลบรารีหลัก (Tech Stack)
- **Framework:** FastAPI
- **การประมวลผลข้อมูลและวิทยาศาสตร์:** `xarray`, `xclim`, `dask`, `numpy`
- **การจัดการภูมิสารสนเทศ (GIS):** `geopandas`, `rioxarray`, `shapely`, `fiona`
- **ฐานข้อมูลและความปลอดภัย:** PostgreSQL / SQLAlchemy, `passlib` (bcrypt), `python-jose` (JWT)

### โครงสร้างไดเรกทอรีหลัก
- `processing/`: แกนกลางทางวิทยาศาสตร์ จัดการคำนวณดัชนี ตัดขอบเขตพื้นที่ และแปลงไฟล์เป็น GeoJSON/JSON
- `services/`: ลอจิกธุรกิจ จัดการงานเบื้องหลัง (Background tasks) และควบคุมการอ่านเขียนไฟล์แบบประหยัดหน่วยความจำ
- `routes/`: จุดเชื่อมต่อ API สำหรับสื่อสารกับ Frontend และจัดการสิทธิ์การเข้าใช้งาน
- `database/`: การตั้งค่าการเชื่อมต่อฐานข้อมูลและโมเดลโครงสร้างข้อมูล (ORM)

### ภาพรวมสถาปัตยกรรม (Architecture Overview)
แผนภาพนี้แสดงถึงโครงสร้างสถาปัตยกรรมแบบโมดูลาร์ของระบบหลังบ้าน ซึ่งมีการแบ่งชั้นการทำงาน (Layers) ออกเป็น 3 ส่วนหลักอย่างชัดเจน เพื่อให้ระบบดูแลรักษาง่ายและมีการแยกสัดส่วนหน้าที่ :

![Backend Class Diagram](images\BackendClassDiagram.png)

1. **ชั้น API Routes (`routes/`):** ด่านหน้ารับคำสั่งจากผู้ใช้งาน จัดการสิทธิ์การเข้าถึง (Authentication) และส่งต่อคำสั่งไปยังระบบหลังบ้าน
2. **ชั้น Service (`services/`):** ผู้ควบคุมลอจิกธุรกิจ (Orchestrator) คอยบริหารจัดการคิวงานเบื้องหลัง (Background Tasks) จัดการไฟล์อย่างปลอดภัย และควบคุม Metadata
3. **ชั้น Processing (`processing/`):** แกนกลางประมวลผลทางวิทยาศาสตร์ รับหน้าที่ประมวลผลหนักทั้งหมด เช่น การตัดขอบเขตเชิงพื้นที่ คำนวณดัชนีภูมิอากาศ และสร้างไฟล์แผนที่ GeoJSON

### กระบวนการทำงานหลัก (Core Workflows)

เพื่อให้เข้าใจถึงสถาปัตยกรรมแบบ Asynchronous และการจัดการหน่วยความจำของระบบหลังบ้าน แผนภาพ Sequence Diagram ด้านล่างนี้จะแสดงลำดับการไหลของข้อมูลใน 2 กระบวนการที่สำคัญที่สุด

#### 1. กระบวนการอัปโหลดและรวมชุดข้อมูล (Data Ingestion & Merging Flow)
กระบวนการนี้แสดงให้เห็นว่าระบบหลังบ้านรับมือกับชุดข้อมูลขนาดใหญ่อย่างไร โดยใช้การอัปโหลดแบบ **Chunked Uploads** เพื่อป้องกัน RAM เต็ม และใช้เทคนิค **Lazy Loading** ผ่าน Dask เพื่อสตรีมข้อมูลในการรวมไฟล์

![Backend Sequence Diagrams 1](images\BackendSequenceDiagrams1.png)

#### 2. กระบวนการคำนวณดัชนีและสร้างแผนที่ (Index Calculation & Map Export Flow)
กระบวนการนี้เป็นหัวใจหลักทางวิทยาศาสตร์ของระบบ แสดงให้เห็นตั้งแต่การนำ Shapefile มาลดทอนรายละเอียดเพื่อสร้าง Cache (ช่วยให้ Frontend โหลดเร็วขึ้น) ตามด้วยการคำนวณดัชนีด้วย `xclim` (แยกรันรายปีและรายเดือน) และวนลูปสร้างแผนที่/อนุกรมเวลาในระดับประเทศและระดับจังหวัด

![Backend Sequence Diagrams 2](images\BackendSequenceDiagrams2.png)

# โมดูลการประมวลผล (processing/)
ไปป์ไลน์ประมวลผลอัตโนมัติแบบครบวงจร ตั้งแต่จัดมาตรฐานและรวมไฟล์ข้อมูลดิบ คำนวณดัชนีและสถิติเชิงพื้นที่ ไปจนถึงส่งออกผลลัพธ์พร้อมแสดงผลบน Frontend

## processing/clipping.py
ตัดขอบเขตเชิงพื้นที่ของข้อมูลสภาพภูมิอากาศแบบหลายมิติด้วย `rioxarray` และ `geopandas` ให้พอดีกับขอบเขตเป้าหมาย พร้อมคำนวณค่าเฉลี่ยเชิงพื้นที่แบบถ่วงน้ำหนักสำหรับรูปทรงที่ไม่สม่ำเสมอ
### Functions
- **resolve_country_names :** ตรวจสอบและคืนค่ารายชื่อเรียกอื่นของประเทศ เพื่อใช้จับคู่ข้อมูลกับ Shapefile ได้ถูกต้อง
- **prep_for_rio :** เตรียมตัวแปร `xarray.DataArray` สำหรับประมวลผล กำหนดแกน X/Y และระบบพิกัด WGS84 (EPSG:4326)
- **clip_to_shape :** ตัดขอบเขตชุดข้อมูลให้พอดีกับรูปทรง Polygon ของ Shapefile ที่กำหนด
- **get_spatial_weights :** สร้างเมทริกซ์ค่าน้ำหนักจากสัดส่วนพื้นที่ทับซ้อนระหว่างกริดกับรูปทรงภูมิศาสตร์จริง
- **calc_weighted_mean :** คำนวณค่าเฉลี่ยแบบถ่วงน้ำหนักด้วยสมการ Σ(Value * Area) / Σ(Area) เพื่อความแม่นยำตามแนวชายแดน

## processing/export_maps.py
สร้างชั้นข้อมูลแผนที่เชิงพื้นที่โดยคำนวณค่าสถิติ (ค่าเฉลี่ยและแนวโน้ม) และแปลงข้อมูลเป็น GeoJSON FeatureCollection เพื่อวาดแผนที่บน Frontend
### Functions
- **export_actual_maps_xesmf :** คำนวณค่าเฉลี่ยตามช่วงเวลา สร้างพิกัดกริดอย่างแม่นยำด้วย `cf_xarray` และส่งออกเป็น GeoJSON แบบกริด
- **export_trend_map_xesmf :** ทดสอบสถิติ Mann-Kendall แบบจุดต่อจุด  เพื่อหาแนวโน้มระยะยาวและส่งออกเป็น GeoJSON แบบกริด
- **export_actual_map_shapefile :** หาค่าเฉลี่ยจากอนุกรมเวลาของแต่ละจังหวัด และจับคู่ลง Polygon ของจังหวัดเพื่อส่งออกเป็น GeoJSON
- **export_trend_map_shapefile :** ทดสอบหาแนวโน้มด้วย Mann-Kendall จากอนุกรมเวลาของแต่ละจังหวัด และส่งออกพร้อมขอบเขตเรขาคณิต

## processing/export_timeseries.py
สกัดและยุบรวมข้อมูลมิติพื้นที่ตามแกนเวลาให้เป็น 1 มิติ ส่งออกเป็นไฟล์ JSON เพื่อแสดงกราฟเส้นหรือแท่ง (รายปีและรายเดือน)
### Functions
- **export_yearly_timeseries :** ยุบรวมหาค่าเฉลี่ยเชิงพื้นที่ จัดกลุ่มตามปี และส่งออกข้อมูลรายปีเป็น JSON
- **export_seasonal_cycle :** หาค่าเฉลี่ยเชิงพื้นที่โดยคงความละเอียดรายเดือน ข้ามค่าว่าง เพื่อส่งออกอนุกรมเวลาแบบต่อเนื่อง

## processing/export_preview.py
ตัวควบคุมไปป์ไลน์สร้างข้อมูลล่วงหน้า (Preview) อัตโนมัติ ตั้งแต่การตัดขอบเขต จนถึงส่งออกแผนที่และอนุกรมเวลาระดับประเทศและจังหวัด
### Functions
- **export_preview_all :** ควบคุมการทำงานดังนี้: 
1. โหลด Shapefile 
2. ตัดขอบเขตเพื่อลดการใช้ RAM 
3. แปลงความถี่ข้อมูลเป็นรายปีและรายเดือน 
4. สร้างแผนที่แบบ Grid และ Polygon พร้อมอนุกรมเวลา

## processing/indices.py
แกนหลักในการคำนวณดัชนีทั้งหมดด้วย `xclim` ตามมาตรฐานสากล ที่เกี่ยวข้องทั้งอุณหภูมิ น้ำฝน และดัชนี SPI (ภัยแล้งและน้ำท่วม) และอัลกอริธึมตรวจจับเหตุการณ์ภัยแล้งและน้ำท่วม
### Functions
- **slice_baseline :** ตัดแบ่งข้อมูลตามช่วงเวลาอ้างอิง (Baseline) อย่างปลอดภัยให้อยู่ในขอบเขตที่มีอยู่จริง
- **compute_pr_percentiles & compute_temp_percentiles :** คำนวณค่าเปอร์เซ็นไทล์ในอดีต สำหรับใช้หาดัชนีสภาพอากาศสุดขั้ว
- **sdii, rx1day, txx, spi, ฯลฯ :** ฟังก์ชันเรียกใช้ `xclim` เพื่อคำนวณดัชนีตามระเบียบวิธีมาตรฐาน
- **event_characteristics_annual_continuous :** สแกนอนุกรมเวลา SPI แบบต่อเนื่อง เพื่อหาความถี่ ระยะเวลา จุดสูงสุด และความรุนแรงของภัยแล้ง/น้ำท่วม จัดกลุ่มตามปี
- **calc_event_maps :** ใช้ `xarray.apply_ufunc` กับ Dask เพื่อคำนวณหาลักษณะเหตุการณ์พร้อมกันทุกพิกเซล
- **convert_temperature_unit :** ตรวจสอบและแปลงหน่วยอุณหภูมิจากเคลวิน (Kelvin) เป็นเซลเซียส (°C) อัตโนมัติ
- **calculate_all_indices :** ผู้ควบคุมสั่งคำนวณดัชนีที่ถูกเลือกทั้งหมด จัดการ Baseline และเก็บผลลัพธ์ลง `xarray.Dataset` 

## processing/overlay.py
ตกแต่งความเรียบร้อยของข้อมูลเชิงพื้นที่ ตัดกริดที่ล้นออกนอกขอบเขตประเทศทิ้ง
### Functions
- **overlay_with_shapefile :** นำไฟล์ GeoJSON ไปหาพื้นที่ทับซ้อน (Intersection) กับ Shapefile ตัดขอบให้แนบสนิท ลดความละเอียดจุดพิกัด และบันทึกทับไฟล์เดิมพร้อมคง Metadata ไว้

## processing/merge_datasets.py
รวมไฟล์ NetCDF เป็น `xarray.Dataset` ก้อนเดียว ป้องกันปัญหา RAM เต็มด้วยเทคนิค Dask Chunking และเขียนลงดิสก์ชั่วคราว
### Functions
- **save_dataset_to_netcdf :** บันทึกไฟล์ชั่วคราว โดยบังคับสตรีมทีละ Chunk (`compute=True`) เพื่อให้การใช้ RAM คงที่
- **_infer_and_fill_gaps :** เรียงลำดับเวลา ตรวจสอบความถี่ และเติมเต็มช่องโหว่เวลา (Gap filling) อัตโนมัติ
- **merge_attribute_mode :** รวมข้อมูลที่พื้นที่และเวลาตรงกัน แต่เก็บตัวแปรต่างชนิดกัน
- **merge_time_mode :** ต่อไฟล์ตัวแปรเดียวกันที่ถูกแบ่งตามเวลาเข้าด้วยกันแบบขนาน
- **merge_mixed_mode :** รวมไฟล์กรณีที่ซับซ้อน โดยต่อแกนเวลาของแต่ละตัวแปรให้เสร็จก่อน แล้วจึงนำทุกตัวแปรมารวมกัน

## processing/pipeline.py
ผู้ควบคุมหลักของเส้นทางกระบวนการทั้งหมด (เตรียมข้อมูล, ตัดพื้นที่, คำนวณ, ส่งออก) เป็นไปป์ไลน์อัตโนมัติเชื่อมกับ API
### Functions
- **generate_all :** ไปป์ไลน์ทำงานหลักเมื่ออัปโหลดข้อมูล: 
1. โหลดข้อมูล 
2. ตัดขอบเขตประเทศ 
3. คำนวณดัชนีรายปี/รายเดือน 
4. วนลูปสร้างแผนที่และอนุกรมเวลาระดับประเทศและจังหวัด
- **generate_custom_map_pipeline :** ไปป์ไลน์สร้างแผนที่แบบ On-demand (เช่น กดดูช่วงปีที่ยังไม่เคยสร้าง) โดยจะตรวจสอบในแคชก่อน หากไม่มีจึงคำนวณเฉพาะดัชนีและขอบเขตที่ขอก่อนทำ Overlay

## processing/preprocessing.py
ทำความสะอาดข้อมูล (Data Cleaning) แปลงชื่อตัวแปร พิกัด และหน่วยวัดให้เป็นมาตรฐานเดียวกันก่อนนำไปประมวลผล
### Functions
- **normalize_var_name :** แปลงชื่อตัวแปรให้เป็นชื่อมาตรฐานตามตาราง (เช่น "precip" ให้เป็น "pr")
- **standardize_coords :** แปลงชื่อพิกัดให้เป็นมาตรฐานสากล (เช่น "lat" ให้เป็น "latitude")
- **ensure_pr_unit :** แปลงหน่วยปริมาณน้ำฝนให้เป็นปริมาณสะสมต่อวัน ("mm/day") อัตโนมัติ
- **ensure_temperature_unit :** แปลงหน่วยอุณหภูมิเป็นองศาเซลเซียส (°C) อัตโนมัติ
- **load_dataset :** เปิดไฟล์ NetCDF/GRIB หรือ CSV และเรียกใช้ฟังก์ชันมาตรฐานทั้งหมดด้านบนก่อนส่งต่อชุดข้อมูล

## processing/upload_validation.py
ด่านตรวจไฟล์อัปโหลดแบบ "Lazy" อ่านเฉพาะ Metadata เบาๆ เพื่อวิเคราะห์รูปแบบการรวมไฟล์และความเข้ากันได้
### Functions
- **filter_dataset_vars :** ลบตัวแปรที่ไม่ใช้ออก เก็บเฉพาะตัวแปรที่ระบบอนุญาต เพื่อลดภาระหน่วยความจำ
- **get_spatial_resolution :** สกัดความละเอียดพื้นที่ (Resolution) โดยหาจาก `rioxarray`, Metadata (ACDD) หรือคำนวณจากพิกัดตามลำดับ
- **inspect_file :** ดึง Metadata สำคัญ (ขนาดไฟล์, ตัวแปร, พิกัด, เวลา) โดยไม่ดึงข้อมูลอาเรย์เข้า RAM
- **detect_mode :** วิเคราะห์รูปแบบการรวมไฟล์ว่าเป็นแบบ Time, Attribute, หรือ Mixed Mode
- **validate_compatibility :** ตรวจสอบไฟล์ในกลุ่มว่ารวมกันได้หรือไม่ ป้องกันข้อผิดพลาดของปฏิทินและความละเอียดพิกัด

# ระบบจัดการ API Routes และ Endpoints (routes/)
ศูนย์รวม Router (FastAPI) เป็นสะพานเชื่อมระหว่าง Frontend และ Backend จัดการ HTTP Requests การยืนยันตัวตน และกระจายงานเบื้องหลัง

## routes/auth_routes.py
จัดการความปลอดภัยและยืนยันตัวตน (Authentication) เข้ารหัสผ่านด้วย bcrypt และออก JWT สำหรับควบคุมสิทธิ์การใช้งาน
### Functions
- **login_user :** ตรวจสอบรหัสผ่าน หากถูกต้องจะคืนค่า JWT Access Token ที่แนบ Role ผู้ใช้งาน
- **Helper Functions :** ประกอบด้วย `verify_password` และ `create_access_token`
- **Pydantic Schemas :** โมเดลตรวจสอบโครงสร้างข้อมูลที่ส่งเข้ามา (UserCreate, UserLogin, TokenResponse)

## routes/dataset_routes.py
API Router หลักสำหรับวงจรชีวิตชุดข้อมูล การคำนวณดัชนี สร้างแผนที่อินเทอร์แอกทีฟ และบริหารพื้นที่เซิร์ฟเวอร์
### Functions
- **การจัดการไฟล์ดิบ :** อัปโหลด เรียกดู และลบไฟล์ในสล็อตชั่วคราว
- **process_selection :** รับคำสั่งส่งไปรัน Background Task เพื่อรวมไฟล์ดิบ
- **calculate_indices_from_slot :** สั่งคำนวณดัชนีใน Background Task ตามเงื่อนไข Baseline และพื้นที่
- **generate_map_endpoint :** สั่งสร้างแผนที่แบบ On-demand (Synchronous) ให้ Frontend รอรับผลทันที
- **get_dataset_metadata :** ดึงข้อมูลไฟล์ `metadata.json` ส่งคืนตัวแปร ช่วงปี และพิกัดที่มีอยู่
- **การจัดการ Shapefile :** อัปโหลดไฟล์พื้นที่ ดึงรายชื่อคอลัมน์ และเรียกดูไฟล์ส่วนตัว/ส่วนกลาง
- **การลบข้อมูล :** ลบชุดข้อมูลหรือ Workspace ได้อย่างปลอดภัย

# บริการระบบหลังบ้านและการจัดการไฟล์ (services/)
ศูนย์กลางเชื่อมต่อ API บริหารจัดการไฟล์ ควบคุมงานเบื้องหลังอย่างปลอดภัย จัดการ Metadata และวิเคราะห์ Shapefile แบบประหยัด RAM ก่อนคำนวณทางวิทยาศาสตร์

## services/dataset_clip.py
ตัดขอบเขตข้อมูลจาก NetCDF ดิบ ทั้งแกนพื้นที่และเวลาแบบ Lazy Evaluation เพื่อกันเซิร์ฟเวอร์ RAM เต็ม
### Functions
- **standardize_coords :** ปรับชื่อพิกัดให้เป็นมาตรฐานสากล
- **get_smart_slice :** ตรวจสอบทิศทางการเรียงพิกัด (น้อยไปมาก หรือ มากไปน้อย) เพื่อสร้างคำสั่งตัด (Slice) ได้ถูกต้อง
- **core_process_file :** ตัดข้อมูล แปลงหน่วย ลบตัวแปรที่ไม่ใช้ และเขียนลงไฟล์ NetCDF4 ชั่วคราว
- **process_and_clip :** วนลูปอ่านและตัดไฟล์ทั้งหมด พร้อมคืนพื้นที่ RAM ด้วยการเรียก Garbage Collector ทันทีหลังจบแต่ละไฟล์

## services/dataset_merge.py
รวมไฟล์ชั่วคราวที่ถูกตัดขอบเขตแล้วเข้าด้วยกัน จัดการลบไฟล์ขยะเพื่อคืนพื้นที่ว่างให้ระบบ
### Functions
- **prepare_merged_file_for_calculation :** ตรวจสอบโหมด นำข้อมูลมาต่อกันจนได้ไฟล์ `merged.nc` แล้วย้ายไฟล์ไปยังที่เก็บผลลัพธ์ `output/` หลักพร้อมลบไฟล์ชั่วคราว

## services/dataset_metadata.py
สแกนหาข้อมูล `merged.nc` เพื่อสร้าง `metadata.json` ให้ Frontend ตรวจสอบตัวแปรที่มีให้เลือกใช้งาน
### Functions
- **is_valid_name :** คัดกรองและลบค่าชื่อที่เป็น "unknown" หรือค่าว่าง
- **read_metadata_json :** โหลดและอ่านไฟล์ `metadata.json` เป็น Python Dictionary
- **get_dataset_metadata_merged :** ดึงตัวแปร ขอบเขตพื้นที่ จำนวนปี ปฏิทิน และตรวจสอบความละเอียด (ระบุกริดไม่สม่ำเสมอด้วย "~")

## services/dataset_paths.py
กำหนดเส้นทางไฟล์ทั้งหมดในระบบ จัดระเบียบแยกข้อมูลผู้ใช้งาน
### Functions
- **get_raw_path :** คืนค่าที่อยู่เก็บไฟล์อัปโหลดดิบ
- **get_processed_path :** คืนค่าที่อยู่เก็บไฟล์ที่ผ่านการตัดขอบเขต
- **get_dataset_output_dir :** คืนค่าที่อยู่เก็บผลลัพธ์สุดท้าย
- **get_user_shapefile_dir & get_global_shapefile_dir :** จัดการเส้นทาง Shapefile ส่วนตัวและส่วนกลาง
- **get_shapefile_path :** ค้นหาไฟล์ `.shp`/`.geojson` โดยไล่หาจากโฟลเดอร์ผู้ใช้ก่อนส่วนกลาง

## services/preview_service.py
ทำงานเบื้องหลังเพื่อสร้างข้อมูลพรีวิวสำหรับตัวแปรพื้นฐาน ให้ผู้ใช้ตรวจสอบภาพรวมทันทีที่รวมไฟล์เสร็จ
### Functions
- **run_preview_visualization :** ตรวจสอบความซ้ำซ้อน หากยังไม่มีให้โหลด `merged.nc` แล้วสั่งไปป์ไลน์สร้าง JSON/GeoJSON ขึ้นมา

## services/shapefile_services.py
ชุดเครื่องมืออ่านข้อมูล Shapefile แบบประหยัด RAM
### Functions
- **detect_region_columns :** ใช้ `fiona` เปิดอ่านเฉพาะ Schema โครงสร้าง กรองหาคอลัมน์ที่เป็นข้อความ แล้วแนะนำคอลัมน์ที่น่าจะเป็นชื่อพื้นที่อัตโนมัติ

## services/dataset_service.py
เป็นศูนย์กลางบริการหลักของระบบหลังบ้าน ทำหน้าที่จัดการการอัปโหลดไฟล์และชุดข้อมูล สั่งรันงานประมวลผลเบื้องหลัง และอัปเดต Metadata
### Functions
- **save_raw_files & upload_and_validate_shapefile :** อัปโหลดไฟล์ (ข้อมูลดิบ/Shapefile) แบบทีละส่วน ป้องกัน RAM กระชาก บังคับใช้ขีดจำกัดขนาดไฟล์ พร้อมระบบแตกไฟล์ Zip และตรวจสอบองค์ประกอบให้ครบถ้วน (.shp, .shx, .dbf, .prj) 
- **update_metadata_json :** อัปเดต `metadata.json` นำข้อมูลใหม่ไปผสานกับของเดิมแบบไม่ซ้ำซ้อน และเรียงลำดับดัชนี
- **run_async_processing :** งานประมวลผลหลังอัปโหลด คุมคิวตัดขอบเขตไฟล์ รวมไฟล์ ดึง Metadata และตามลบโฟลเดอร์ไฟล์ชั่วคราวทิ้งทันที
- **run_async_calculation :** งานคำนวณดัชนีภูมิอากาศ ลดทอนรายละเอียดพิกัด Shapefile เพื่อสร้าง `boundary.geojson` ขนาดเล็กเป็น Cache ช่วยให้แผนที่โหลดรวดเร็วขึ้น จากนั้นสั่งรันไปป์ไลน์คำนวณ
- **generate_on_demand_map :** สั่งสร้างแผนที่เฉพาะช่วงปีตามคำขอ โดยระบบจะบล็อกการตอบกลับจนกว่าไฟล์จะสร้างเสร็จ เพื่อให้ Frontend ดึงไปแสดงผลได้ทันที
